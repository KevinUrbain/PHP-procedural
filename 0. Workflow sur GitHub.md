# Workflow sur GitHub

# LE WORKFLOW COMPLET DE COLLABORATION SUR GITHUB

## ⚙️ 1. CONFIGURATION INITIALE (une seule fois)

### **A. Configuration de Git**

```bash
# Identité
git config --global user.name "Ton Nom"
git config --global user.email "ton.email@example.com"

# Éditeur par défaut
git config --global core.editor "code --wait"

# Vérification
git config --list
```

### **B. Configuration SSH (optionnel mais recommandé)**

```bash
# Génère une clé SSH
ssh-keygen -t ed25519 -C "ton.email@example.com"
# Appuie sur Entrée 3 fois

# Copie la clé publique
cat ~/.ssh/id_ed25519.pub
# Colle le résultat dans GitHub → Settings → SSH Keys
```

### **C. Structure de dossiers professionnelle**

```bash
# Crée ton dossier de projets
mkdir -p ~/Documents/Projets-IFAPME
# Ou pour Windows avec Laragon :
cd /c/laragon/www
```

## 🎬 2. DÉMARRAGE D'UN NOUVEAU PROJET

### **CAS 1 : Tu rejoins un projet existant (collaborateur)**

```bash
# 1. Va dans ton dossier de projets
cd /c/laragon/www

# 2. Clone le projet
git clone https://github.com/auteur/projet.git
cd projet

# 3. Vérifie la structure
ls -la
# Tu dois voir : .git/ (IMPORTANT)

# 4. Regarde les branches existantes
git branch -a

# 5. Synchronise avec la branche principale
git checkout main
git pull origin main

# 6. Crée TA branche de travail
git checkout -b feature/ton-nom-fonctionnalite
# Exemples : 
# - backend-Kevin
# - feature/auth-system
# - fix/bug-login

# 7. Ouvre ton éditeur
code .
```

### **CAS 2 : Tu contribues sans être collaborateur (fork)**

```bash
# 1. Sur GitHub : clique "Fork" sur le repo original
# Maintenant tu as : github.com/TON-NOM/projet

# 2. Clone TON fork
git clone https://github.com/TON-NOM/projet.git
cd projet

# 3. Ajoute le repo original comme "upstream"
git remote add upstream https://github.com/auteur-original/projet.git

# 4. Vérifie les remotes
git remote -v
# origin → ton fork (tu peux push)
# upstream → repo original (tu peux pull)

# 5. Crée ta branche
git checkout -b feature/ma-contribution

# 6. Code...
```

## 💼 3. WORKFLOW QUOTIDIEN

### **🌅 DÉBUT DE JOURNÉE**

```bash
# 1. Va dans ton projet
cd /c/laragon/www/FashionFamily

# 2. Vérifie l'état
git status

# 3. Synchronise avec main (récupère les nouveautés)
git checkout main
git pull origin main

# 4. Retourne sur ta branche
git checkout backend-Kevin

# 5. Intègre les nouveautés de main
git merge main
# OU (plus propre, garde historique linéaire) :
git rebase main

# 6. Si conflits, résous-les (voir section 5)

# ✅ Maintenant tu es à jour, tu peux coder !
```

### **💻 PENDANT LA JOURNÉE**

```bash
# Tu codes, tu codes, tu codes...

# Commits réguliers (plusieurs fois par jour)
# Règle : 1 commit = 1 modification logique

# Exemple 1 : Tu as ajouté une fonction
git add src/Auth.php
git commit -m "feat: ajout méthode de validation email"

# Exemple 2 : Tu as corrigé un bug
git add pages/login.php
git commit -m "fix: correction erreur validation mot de passe"

# Exemple 3 : Tu as refactorisé
git add src/Controllers/AuthController.php
git commit -m "refactor: extraction logique auth dans contrôleur"

# Push régulièrement (backup + visibilité pour l'équipe)
git push origin backend-Kevin

# Si première fois :
git push -u origin backend-Kevin
# -u = set-upstream (définit la branche distante par défaut)
```

### 🌙 FIN DE JOURNÉE

```bash
# 1. Vérifie qu'il n'y a rien en attente
git status

# 2. Si des fichiers modifiés non commités
git add .
git commit -m "wip: travail en cours sur [fonctionnalité]"
# wip = Work In Progress

# 3. Push final
git push origin backend-Kevin

# 4. (Optionnel) Laisse un message à l'équipe
# "J'ai avancé sur le système d'auth, c'est sur ma branche backend-Kevin"
```

## 📬 4. PULL REQUESTS (PR)

### **Quand créer une Pull Request ?**

✅ Ta fonctionnalité est **terminée et testée**
✅ Ton code **compile/fonctionne** sans erreurs
✅ Tu as **commité et pushé** tous tes changements
✅ Ta branche est **à jour** avec `main`

### **Comment créer une Pull Request ?**

```bash
# 1. Assure-toi que ta branche est pushée
git push origin backend-Kevin

# 2. Va sur GitHub → onglet "Pull requests"
# 3. Clique "New pull request"

# 4. Configure la PR :
# base: main ← head: backend-Kevin

# 5. Remplis le formulaire
```

### **Template de Pull Request professionnelle**

```markdown
## 📝 Description
Implémentation complète du système d'authentification en POO.

## ✨ Fonctionnalités ajoutées
- ✅ Classe AuthController pour gérer login/register/logout
- ✅ Modèle User avec validation
- ✅ Hashage bcrypt des mots de passe
- ✅ Protection CSRF sur les formulaires
- ✅ Sessions sécurisées

## 📁 Fichiers modifiés
- `src/Controllers/AuthController.php` (nouveau)
- `src/Models/User.php` (nouveau)
- `pages/login.php` (refactorisé)
- `pages/register.php` (refactorisé)

## 🧪 Tests effectués
- [x] Login avec identifiants valides → OK
- [x] Login avec mauvais mot de passe → Message d'erreur
- [x] Register avec email déjà existant → Message d'erreur
- [x] Sessions persistent après refresh → OK
- [x] Logout détruit la session → OK

## 📸 Screenshots (si pertinent)
[Capture d'écran de la page de login]

## ⚠️ Points d'attention

- La sécurité du hashage bcrypt
- La gestion des erreurs

```

### **Après la création de la PR**

```bash
# 1. Attends les commentaires de review

# 2. Si des modifications sont demandées :
git checkout backend-Kevin
# ... fais les corrections ...
git add .
git commit -m "review: correction selon commentaires de @S-Sumael"
git push origin backend-Kevin
# Les changements s'ajoutent AUTOMATIQUEMENT à la PR

# 3. Réponds aux commentaires sur GitHub
# "✅ Corrigé !" ou "Bonne idée, modifié"

# 4. Quand tout est validé, le mainteneur merge

# 5. APRÈS le merge, nettoie ta branche
git checkout main
git pull origin main  # Récupère le merge
git branch -d backend-Kevin  # Supprime en local
git push origin --delete backend-Kevin  # Supprime sur GitHub (optionnel)
```

## ⚔️ 5. GESTION DES CONFLITS

### **Quand surviennent les conflits ?**

- Toi et un collègue avez modifié **la même ligne** du même fichier
- Tu as renommé/supprimé un fichier que quelqu'un a modifié
- Deux branches ont divergé significativement

### **Résolution étape par étape**

```bash
# Situation : Tu essaies de merger main dans ta branche
git checkout backend-Kevin
git merge main

# Git répond :
# CONFLICT (content): Merge conflict in pages/login.php
# Automatic merge failed; fix conflicts and then commit the result.

# 1. Vérifie quels fichiers sont en conflit
git status
# Unmerged paths:
#   both modified:   pages/login.php

# 2. Ouvre le fichier en conflit
code pages/login.php

# 3. Cherche les marqueurs de conflit
<<<<<<< HEAD
// TON code (sur backend-Kevin)
function login($email, $password) {
    // ta version
}
=======
// Code de la branche main
function authenticate($user, $pass) {
    // version du collègue
}
>>>>>>> main

# 4. Décide quoi garder (3 options)
# Option A : Garder TON code uniquement
function login($email, $password) {
    // ta version
}

# Option B : Garder le code de main uniquement
function authenticate($user, $pass) {
    // version du collègue
}

# Option C : Combiner les deux (souvent la meilleure solution)
function login($email, $password) {
    // ta version améliorée avec les ajouts du collègue
}

# ⚠️ SUPPRIME TOUJOURS LES MARQUEURS <<<<<<< ======= >>>>>>>

# 5. Sauvegarde le fichier

# 6. Marque le conflit comme résolu
git add pages/login.php

# 7. Finalise le merge
git commit -m "merge: résolution conflit dans pages/login.php"

# 8. Push
git push origin backend-Kevin

# ✅ Conflit résolu !
```

### **Conseils pour éviter les conflits**

```bash
✅ Synchronise TOUS LES JOURS avec main
✅ Communique avec l'équipe sur qui travaille sur quoi
✅ Fais des petites Pull Requests fréquentes
✅ Ne laisse pas ta branche diverger pendant des semaines
✅ Utilise git rebase au lieu de merge pour un historique propre
```

## 🛠️ 6. COMMANDES ESSENTIELLES

### **Navigation et statut**

```bash
# Afficher l'état actuel
git status

# Afficher l'historique
git log --oneline -10  # 10 derniers commits
git log --graph --all  # Graphique des branches

# Voir les différences
git diff                    # Modifications non staged
git diff --staged           # Modifications staged
git diff main..ta-branche   # Différences entre branches
```

### **Branches**

```bash
# Lister les branches
git branch              # Locales
git branch -a           # Toutes (locales + distantes)

# Créer et basculer
git checkout -b nouvelle-branche   # Crée et bascule
git switch nouvelle-branche        # Alternative moderne

# Basculer de branche
git checkout main
git switch main  # Alternative moderne

# Supprimer une branche
git branch -d nom-branche      # Si mergée
git branch -D nom-branche      # Force (si pas mergée)
```

### **Synchronisation**

```bash
# Récupérer les changements (sans merger)
git fetch origin

# Récupérer et merger
git pull origin main

# Envoyer les changements
git push origin ta-branche

# Premier push d'une nouvelle branche
git push -u origin ta-branche
```

### **Commits**

```bash
# Ajouter des fichiers
git add fichier.php          # Un fichier
git add .                    # Tous les fichiers
git add src/                 # Un dossier

# Commiter
git commit -m "feat: message"

# Modifier le dernier commit (avant push)
git commit --amend -m "nouveau message"

# Annuler le dernier commit (garder les changements)
git reset --soft HEAD~1

# Annuler le dernier commit (supprimer les changements)
git reset --hard HEAD~1  # ⚠️ DANGEREUX
```

### **Annulations et corrections**

```bash
# Annuler les modifications d'un fichier non commité
git checkout -- fichier.php
git restore fichier.php  # Alternative moderne

# Retirer un fichier du staging
git reset HEAD fichier.php
git restore --staged fichier.php  # Alternative moderne

# Revenir à un commit précédent
git checkout abc1234  # abc1234 = hash du commit

# Créer une nouvelle branche depuis un commit
git checkout -b nouvelle-branche abc1234
```

### **Stash (mise de côté temporaire)**

```bash
# Situation : Tu dois changer de branche mais ton travail n'est pas prêt

# Mettre de côté
git stash save "Description du travail en cours"

# Changer de branche et travailler...
git checkout autre-branche

# Revenir et récupérer le travail
git checkout ta-branche
git stash pop  # Récupère et supprime du stash

# Lister les stash
git stash list

# Appliquer un stash spécifique sans le supprimer
git stash apply stash@{0}
```

```bash
git checkout main

```

### **Workflow B : Gros projet avec équipe**

```bash
# Structure :
main (production)
  └─ develop (développement)
      ├─ feature/auth (toi)
      ├─ feature/dashboard (collègue 1)
      └─ bugfix/login (collègue 2)

# Ton workflow :
git checkout develop
git pull origin develop
git checkout -b feature/auth
# ... code ...
git push origin feature/auth
# Pull Request : feature/auth → develop

# Quand develop est stable :
# Le lead merge develop → main
```

### **Workflow C : Contribution open-source**

```bash
# 1. Fork sur GitHub
# 2. Clone TON fork
git clone https://github.com/TON-NOM/projet.git
git remote add upstream https://github.com/original/projet.git

# 3. Crée ta branche depuis main à jour
git fetch upstream
git checkout main
git merge upstream/main

# 4. Code et push vers TON fork
git push origin fix/typo-readme

# 5. Pull Request depuis TON fork vers le repo original
# Base: original/projet:main ← Head: TON-NOM/projet:fix/typo-readme
```

## 📊 CONVENTIONS DE MESSAGES DE COMMIT

### **Format standard : Conventional Commits**

```bash
<type>: <description courte>

[corps optionnel]

[footer optionnel]
```

### **Types courants**

```bash
feat:      Nouvelle fonctionnalité
fix:       Correction de bug
docs:      Documentation uniquement
style:     Formatage (espaces, virgules, pas de changement de code)
refactor:  Refactorisation (ni bug ni feature)
test:      Ajout/modification de tests
chore:     Tâches diverses (config, build, etc.)
perf:      Amélioration des performances
```

### **Exemples concrets**

```bash
# ✅ Bons exemples
feat: ajout du système d'authentification OAuth
fix: correction validation email dans formulaire register
docs: mise à jour README avec instructions d'installation
refactor: extraction logique métier dans service dédié
test: ajout tests unitaires pour classe User
style: correction indentation fichiers PHP selon PSR-12

# ❌ Mauvais exemples
"update"
"fix"
"changes"
"wip" (sauf en fin de journée)
```

## 🚨 ERREURS COURANTES ET SOLUTIONS

| 🔴 Erreur | ✅ Solution |
| --- | --- |
| `error: failed to push` | `git pull origin ta-branche` puis `git push` |
| `fatal: not a git repository` | Tu n'es pas dans le bon dossier, `cd` jusqu'au dossier avec `.git/` |
| `Your branch is behind` | `git pull origin ta-branche` pour te mettre à jour |
| Commits sur `main` par erreur | `git reset --soft HEAD~1` puis `git checkout -b nouvelle-branche` |
| Force push accidentel | ⚠️ **PRÉVIENS L'ÉQUIPE** immédiatement |
| Conflits incompréhensibles | Demande de l'aide, ne fusionne pas à l'aveugle |

### 🎯 RÉSUMÉ EN 10 COMMANDEMENTS

```bash
1. Ne code JAMAIS directement sur main
2. Crée UNE branche par fonctionnalité
3. Commit RÉGULIÈREMENT avec des messages clairs
4. Pull AVANT de push
5. Synchronise avec main TOUS LES JOURS
6. Fais des Pull Requests PETITES et fréquentes
7. Résous les conflits SOIGNEUSEMENT
8. Communique avec ton ÉQUIPE
9. Test ton code AVANT de push
10. Never force push sur des branches PARTAGÉES
```
